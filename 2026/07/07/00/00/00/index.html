本地大模型 + 私有知识库，是 2026 年最实用的 AI 落地方案之一。不依赖云端 API，数据不出机器，还能针对你的文档精准问答。这篇文章用三块内容把整条链路讲透：RAG 是什么、怎么把文档变成向量数据库、怎么跟 llama.cpp + MiniCPM5 快速接起来。全程本地，零 API 费用。

一、什么是 RAG：原理与效率

1.1 一句话说清楚

RAG（Retrieval-Augmented Generation，检索增强生成） = 先从你的文档里检索相关内容，再把检索到的内容塞进大模型的上下文，让模型基于这些内容回答问题。

打个比方：大模型像是一个聪明的学生，知识渊博但没看过你的课本。RAG 就是考试前临时把你的课本翻到相关页码，摊在他面前——他靠自己的理解力答题，但答案有据可查。

1.2 为什么需要 RAG

大模型有三个先天缺陷，RAG 逐个解决：

大模型的问题 RAG 怎么解决
知识截止：训练数据有截止日期，不知道最新信息 文档随时更新，检索到就能用
幻觉：不知道的事会编造，说得还很自信 检索到的文档作为事实依据，模型”看着答”
不懂你的私有数据：没见过你的内部文档/代码/笔记 你的文档就是知识源，模型不需要”记住”

另一个方案是微调（Fine-tuning），但对比一下：

维度 RAG 微调
数据更新 随时增删文档，即时生效 重新训练，数小时到数天
成本 一次性建库，查询成本极低 训练成本高（GPU 时间 + 数据标注）
可解释性 能看到检索到了哪些文档 黑箱，无法追溯答案来源
适合场景 文档问答、知识库、动态数据 固定风格/领域深度适配
幻觉控制 好，有文档兜底 一般，模型仍可能编造

结论：90% 的”让 AI 懂我的文档”需求，RAG 是正确答案。 微调适合改变模型行为风格，RAG 适合给模型补充知识。

1.3 RAG 的完整原理

RAG 系统分两个阶段：

阶段一：建库（离线，做一次）

阶段二：查询（在线，每次提问）

检索的核心：向量相似度

文本被 Embedding 模型转成高维向量（比如 1024 维浮点数）。语义相近的文本，向量距离也近。检索时计算问题向量和所有文档向量的距离，取最近的 K 个。

常用的距离度量：

  • 余弦相似度（Cosine Similarity）：最常用，衡量方向一致性，不关心向量长度
  • 内积（Inner Product）：归一化后等价于余弦相似度
  • L2 距离（欧氏距离）：衡量绝对位置差异

1.4 RAG 的效率

检索效率

向量数据库的检索复杂度是 O(log N)（近似最近邻搜索，ANN），不是 O(N) 线性扫描。实际体验：

文档量 建库时间 单次检索延迟
100 个文本块 < 1 秒 < 10ms
10,000 个文本块 几十秒 < 20ms
1,000,000 个文本块 数十分钟 < 50ms

对比全文检索（如 Elasticsearch），向量检索在语义匹配上有质的优势——“怎么部署模型”能匹配到”安装步骤”，关键词搜索做不到。

生成效率

RAG 的端到端延迟 = 检索延迟 + 模型推理延迟。检索只占很小一部分（约 5-15%），主要瓶颈在模型生成。

用 MiniCPM5-1B（1B 参数）的推理速度：

硬件 速度（Q4 量化） 端到端延迟（检索+生成 200 token）
CPU（8核 M-series） ~30 tok/s ~7 秒
GPU（RTX 3060 12GB） ~120 tok/s ~2 秒
GPU（RTX 4090 24GB） ~300 tok/s < 1 秒

1B 模型小，推理快，适合做 RAG 的生成端。检索几乎不花时间，体验瓶颈在模型生成速度。

准确率

RAG 的准确率取决于四个因素：

  1. 文档质量——垃圾进垃圾出，OCR 错误/格式混乱会严重影响效果
  2. 分块策略——chunk 太大召回率低，太小丢失上下文（后面详讲）
  3. Embedding 模型——好模型语义理解更准，推荐 mxbai-embed-large 或 bge-m3
  4. Top-K 选择——K 太少漏掉相关内容，太多噪声干扰（通常 K=3~5）

工业实践里，好的 RAG 系统在文档问答任务上能达到 85-95% 的准确率（答案能从检索到的文档中找到依据），远高于纯大模型的 40-60%（全靠模型自己编）。

1.5 RAG 的进阶：不是简单拼接

基础 RAG 就是”检索+拼接+生成”，但生产级 RAG 还需要：

  • 重排（Reranking）：先用向量检索召回 Top-20，再用 Cross-Encoder 精排到 Top-5，准确率显著提升
  • 混合检索：向量检索（语义） + 关键词检索（BM25），取并集，覆盖不同查询类型
  • 查询改写：把用户问题改写成更适合检索的形式（”去年的营收” → “2025年 营业收入 财报”）
  • 多跳检索：复杂问题拆成子问题，分别检索再合并

这篇文章聚焦基础 RAG，把基础跑通再考虑进阶。

二、如何将文档变成 RAG 数据库

2.1 技术选型

组件 推荐选择 理由
Embedding 模型 mxbai-embed-large-v1 开源、支持 llama.cpp 原生推理
向量数据库 ChromaDB 轻量、纯 Python、零配置，适合中小规模
文档解析 LangChain TextLoader 系列 覆盖 PDF/Word/Markdown/HTML 等主流格式
分块工具 RecursiveCharacterTextSplitter 按语义边界递归切分，优于固定长度

如果你追求极致性能或百万级文档，可以换 Qdrant 或 Milvus。但 99% 的个人项目，ChromaDB 够了。

2.3 第一步：文档解析——把各种格式变成纯文本

关键点：metadata 一定要保留。RAG 系统回答时引用来源全靠它。PyPDFLoader 会自动记录页码，DirectoryLoader 会记录文件路径。

2.4 第二步：分块（Chunking）——最影响效果的环节

分块是 RAG 中最容易被忽视、但对效果影响最大的步骤。

为什么要分块

  • 模型上下文有限：不能把整本书塞进去
  • 检索精度：小段文本的向量更聚焦，检索更准。整章的向量”什么都像又什么都不像”
  • 生成质量：塞太多无关内容，模型容易跑偏

分块策略

chunk_size 怎么选

chunk_size 适合场景 优点 缺点
200-300 精确问答（FAQ 式） 检索精准 上下文不足，答案可能不完整
500-800 通用（推荐默认） 平衡精度和上下文 中规中矩
1000-1500 长文档总结、报告分析 上下文完整 检索噪声多，向量不够聚焦

经验值：中文文档用 500 字符 + 50 overlap，英文用 800 字符 + 100 overlap。先跑通再调。

切了等于没切？检查分块质量

如果看到一句话被从中间切断，调整 separators 优先级或增大 chunk_overlap。

三、如何快速与 llama.cpp 结合

3.1 架构总览

llama.cpp 的 llama-server 同时提供 Embedding 和 Chat 两个端点，一个进程搞定两件事。

3.2 第一步：部署 MiniCPM5

获取模型

MiniCPM5-1B 是 OpenBMB 2026 年 5 月发布的 1B 参数模型：

  • 参数量：10.8 亿（约 1GB 显存即可跑 Q4 量化）
  • 上下文：131,072 tokens（128K，业界同级别最长）
  • 架构：标准 LlamaForCausalLM，24 层，GQA（16 Q heads + 2 KV heads）
  • 特性：内置 <think> 模板，支持 Think/No-Think 双模式
  • 能力：在 1B 级别模型中评分第一（Artificial Analysis Intelligence Index 17.9 分）

为什么选 MiniCPM5 做 RAG：

  1. 小：Q4 量化仅 688MB，随便一张显卡甚至纯 CPU 都能跑
  2. 长上下文：128K 上下文意味着检索回来的文档可以塞很多
  3. 够聪明：1B 里的天花板，RAG 场景下不需要复杂推理，够用
  4. 中英双语：中文文档 RAG 不费劲

下载 GGUF

量化版本选择：

量化 文件大小 质量损失 推荐
Q4_K_M 688 MB 极小 ⭐ 首选
Q5_K_M ~800 MB 几乎无 追求质量
Q8_0 1.15 GB 无损 精度优先
F16 2.17 GB 无损 调试用

日常用 Q4_K_M，688MB 跟一张手机照片差不多大。

上下文窗口利用

MiniCPM5 支持 128K 上下文，但实际使用中不必开那么大（内存开销大）。建议：

文档量 建议 -c 值 内存占用（Q4）
< 50 个 chunk 4096 ~1.5GB
50-500 个 chunk 8192 ~2GB
500+ 个 chunk 16384 ~3GB

RAG 每次只塞 Top-K（3-5 个）chunk 进上下文，每个 chunk 约 500 字符 ≈ 250 tokens，5 个 chunk ≈ 1250 tokens。加上系统提示和用户问题，总共不超过 2000 tokens。所以 -c 4096 其实就够了，留余量用 8192。

3.8 完整项目结构

总结：三步走回顾

核心技术决策三个：

  1. 模型选 MiniCPM5-1B——1B 参数、688MB、128K 上下文，本地 RAG 的甜点
  2. 向量库选 ChromaDB——零配置、纯 Python、10 万级文档够用
  3. 推理用 llama.cpp——一个进程同时跑生成和 Embedding，部署最简

最容易踩的坑：分块。chunk_size 太大检索不聚焦，太小上下文断裂。先 500 字符跑通，再根据效果调。

最大的误区：以为 RAG 效果差是模型不行。90% 的情况是检索环节的问题——文档没解析干净、分块不合理、Embedding 模型太弱。先查检索，再查生成。
