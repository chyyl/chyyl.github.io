Supra-Router-51M + llama.cpp Router 模式：用 51M 小模型智能路由多模型

51M 参数的路由模型，决定你的 prompt 该交给 3B 小模型还是 70B 大模型处理。llama.cpp 的 Router 模式让你同时挂多个 GGUF 模型，按需加载卸载。这篇讲清楚两件事：Supra-Router-51M 是什么怎么用，llama.cpp Router 模式怎么配。

一、问题：为什么要模型路由

1.1 现实困境

本地部署 LLM 时，你经常面对一个选择题：

模型大小 优点 缺点
3B / 8B 小模型 快、省内存、能跑在手机上 复杂推理不行、代码能力弱
32B / 70B 大模型 推理强、代码好、回答质量高 慢、吃显存、并发能力差

理想方案：简单问题给小模型，复杂问题给大模型。

但怎么判断一个问题是简单还是复杂？

  • 手写规则？if "代码" in prompt: route_to_big_model —— 太死板，覆盖不了真实场景
  • 用 Embedding 算相似度？需要预定义分类库，维护成本高
  • 用一个 LLM 来判断？这就是 Supra-Router-51M 做的事

1.2 Supra-Router-51M 是什么

Supra-Router-51M 由 SupraLabs（独立研究小组）发布，核心定位：

  • 参数量：51.7M —— 极小，CPU 秒级推理
  • 功能：分析用户 prompt，输出路由决策（走小模型 / 走大模型）
  • 附加信息：领域分类、复杂度评分（1-5）、是否需要数学/代码能力、理由
  • 训练数据：SupraLabs/Prompt-Routing-Dataset（992 条）
  • 协议：MIT 开源
  • 格式：Safetensors（原始）+ GGUF（llama.cpp 可用）

1.3 llama.cpp Router 模式是什么

llama.cpp 的 llama-server 在 2025 年 12 月新增了 Router 模式：

  • 不再绑定单个 GGUF 文件
  • 指定一个模型目录，自动发现所有 GGUF 文件
  • 按需加载：第一次请求某模型时才加载到内存
  • LRU 淘汰：达到最大同时加载数时，自动卸载最久没用的
  • 每个模型独立进程，一个崩溃不影响其他

这两个东西配合使用：Supra-Router 判断路由方向 → llama.cpp Router 模式执行模型切换。

字段 取值 说明

Math True/False 是否需要数学推理
Code True/False 是否需要代码生成
Route small model / big model 最终路由决策
Justification 文本 路由理由

2.2 多任务序列设计

Supra-Router 不是简单的二分类器。它采用 Multi-Task Sequence Generation（多任务序列生成）：

模型在输出 Route 之前，必须先算出 Domain、Complexity、Math、Code。这些前置任务构成了特征激活图，防止权重坍缩，保证路由边界稳定。

这种设计让 51M 的小模型也能做出可靠的路由决策。

2.3 训练细节

  • 基座：51M 参数的预训练模型
  • 微调数据：SupraLabs/Prompt-Routing-Dataset（992 条）
  • 训练轮数：5 Epochs
  • 最佳检查点：Epoch 3（eval_loss: 0.1342）
  • 为防止后期过拟合，在 Epoch 3 后停止训练

2.4 典型路由示例

用户输入 Domain Complexity Math Code Route
“你好” General 1 False False small model
“解释量子纠缠” Physics 3 False False big model
“2+2等于几” Mathematics 1 True False small model
“证明 √2 是无理数” Mathematics 5 True False big model
“写个 Python 脚本批量重命名文件” CS 3 False True big model
“今天天气怎么样” General 1 False False small model
“用 Rust 实现一个 TCP 服务器” CS 5 False True big model

六、性能与成本分析

6.1 Supra-Router 开销

指标 数值
模型大小 ~60MB（Q8_0 GGUF）
推理时间（CPU） 20-50ms
推理时间（GPU） 5-10ms
内存占用 ~200MB
Token 消耗 输入 ~50 tokens，输出 ~50 tokens

路由判断本身的开销极小，相比大模型推理动辄几秒，几乎可以忽略。

6.2 路由节省的成本

假设 1000 次请求的场景：

方案 大模型调用次数 平均延迟 总 GPU 时间
全部走大模型 1000 3s 3000s
全部走小模型 0 0.3s 300s（质量差）
Supra-Router 路由 ~300（30%复杂） 1.1s ~1080s

路由方案在质量和速度之间取得平衡：简单请求秒回，复杂请求走大模型保证质量。

6.3 适合场景

  • 边缘设备：显存有限，不能同时加载多个大模型
  • 多租户服务：不同用户请求复杂度不同
  • 混合云架构：小模型本地跑，大模型走云 API
  • A/B 测试：同时运行多个模型版本，按需切换

八、常见问题

问题 解答
Supra-Router 支持中文吗？ 支持，训练数据含多语言 prompt，但英文准确率更高
能路由到云 API 吗？ 可以，在 Python 网关中把 “big model” 改为调用 OpenAI/Claude API
路由准确率怎样？ 官方说”非常精确，偶尔领域分类会幻觉”，路由决策本身可靠
能自定义路由类别吗？ 当前只输出 small/big 两档，如需更多分类可训练自己的模型
Router 模式支持流式吗？ 支持，在请求中加 "stream": true
多用户并发怎样？ 每个模型独立进程，不同模型可并行处理；同模型内排队
如何监控模型状态？ GET /v1/models 返回每个模型的 loaded/loading/unloaded 状态

总结速查

组件 作用 关键配置
Supra-Router-51M 判断 prompt 该走大模型还是小模型 Task: {prompt}\nAnalysis: 格式输入，temp=0.1
llama.cpp Router 模式 多模型管理，按需加载卸载 --models-dir + --models-max + --models-preset
Python 路由网关 串联两者，用户无感 Supra-Router 判断 → 选模型 → 转发请求
