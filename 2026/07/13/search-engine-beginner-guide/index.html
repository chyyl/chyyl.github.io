你每天用搜索引擎几十次，但有没有想过：输入几个词，为什么 0.3 秒就能从千亿网页中找到答案？这篇从零开始，讲清楚搜索引擎的爬虫抓取、索引构建、排序算法、数据处理全流程，再带你动手写一个迷你搜索引擎。

一、搜索引擎是什么

1.1 定义

搜索引擎是一个系统：用户输入查询词，系统从预先构建的网页索引中快速找出相关页面，按相关性排序后返回结果。

1.2 搜索引擎的三个核心组件

组件 作用 关键技术
爬虫（Crawler） 从互联网抓取网页 URL 队列、去重、礼貌策略、增量爬取
索引器（Indexer） 把网页组织成可快速检索的数据结构 倒排索引、分词、压缩
查询处理器（Query Processor） 接收查询，检索并排序返回 召回、排序算法（PageRank / BM25 / 向量）

1.3 搜索 vs 数据库查询

很多人会问：为什么不直接用 SQL 的 LIKE '%关键词%' 搜索？

对比 SQL LIKE 搜索引擎
速度 全表扫描，O(n) 倒排索引，O(1) 查找
相关性 有或没有，没有排序 按相关性排序
模糊匹配 不支持 支持同义词、拼写纠错
分词 无 中英文分词
规模 百万级 千亿级

二、爬虫：数据从哪来

2.1 爬虫基本原理

爬虫（也叫 Spider / Bot）是一个程序：给定一批种子 URL，不断从网上下载网页，从中提取新的 URL，加入队列继续下载，循环往复。

2.2 核心问题

问题 1：怎么知道哪些页面已经爬过？

用布隆过滤器（Bloom Filter）。它是一个空间效率极高的概率数据结构：

  • 将 URL 通过多个哈希函数映射到一个比特数组
  • 查询时检查比特位：全为 1 → 可能已爬过（有假阳性）；有为 0 → 肯定没爬过
  • 100 亿 URL 只需约 12GB 内存

问题 2：怎么选择下一个要爬的 URL？

不是先来先爬。搜索引擎用优先级队列：

  • 首页质量高的网站优先
  • 更新频繁的页面优先
  • 新发现的 URL 优先级低于已知重要页面

问题 3：怎么避免把对方网站爬崩？

遵守礼貌策略（Politeness）：

  • 同一域名同时只有一个连接
  • 每次请求间隔至少 1-2 秒
  • 遵守 robots.txt 协议（声明哪些页面可以爬）

问题 4：怎么发现新内容？

两种策略：

  • 增量爬取：已知页面定期重新爬，发现变化
  • 深度优先 vs 广度优先：一般用广度优先，先覆盖更多网站

三、倒排索引：搜索引擎的心脏

3.1 正排索引 vs 倒排索引

正排索引（文档 → 词）：

查”倒排索引”需要扫描所有文档 → 太慢。

倒排索引（词 → 文档）：

查”倒排索引”直接得到 [文档1, 文档2] → O(1) 查找。

3.2 倒排索引的完整结构

真实的倒排索引不只记录”哪个文档包含这个词”，还记录很多附加信息：

  • 词频（TF）：这个词在文档中出现几次。出现越多越相关
  • 位置（Position）：出现的位置。用于短语查询和 proximity 搜索
  • 字段（Field）：出现在标题、正文还是 URL 中。标题中出现权重更高

3.5 停用词

有些词出现频率极高但没有实际含义，索引它们浪费空间：

英文：the, is, at, which, on, a, an, of, to, in

中文：的, 了, 是, 在, 有, 和, 就, 也

这些词叫停用词（Stop Words），通常在建索引时直接丢弃。

但注意：短语查询（如 “to be or not to be”）需要保留停用词才能精确匹配，所以现代搜索引擎通常还是索引停用词，只是给它们很低的权重。

四、排序算法：决定你看到什么

找到包含查询词的文档只是第一步，关键是哪些排在前面。

4.1 TF-IDF

最经典的文本相关性评分。

TF（Term Frequency，词频）：查询词在文档中出现越多，越相关。

IDF（Inverse Document Frequency，逆文档频率）：查询词在所有文档中越罕见，越有价值。

“的”出现在几乎所有文档 → IDF 接近 0 → 权重极低
“搜索引擎”只出现在少数文档 → IDF 大 → 权重高

TF-IDF = TF × IDF

4.2 BM25

TF-IDF 的升级版，目前仍是全文搜索的标准排序算法。

BM25 对 TF 做了改进：不是线性增长，而是趋于一个上限。一个词在文档中出现 10 次和 100 次，相关性不应该差 10 倍。

参数：

  • f(qi, D)：词 qi 在文档 D 中的频率
  • |D|：文档长度
  • avgdl：平均文档长度
  • k1：通常 1.2-2.0，控制 TF 增长速度
  • b：通常 0.75，控制文档长度归一化强度

直觉理解：

  • 文档越长，同样的词频权重越低（可能只是因为文档长才出现多）
  • 词频增长到一定程度后不再增加权重（饱和效应）
  • 相比 TF-IDF，BM25 对长文档更公平

Elasticsearch / Lucene 默认使用 BM25 排序。

4.3 PageRank

TF-IDF 和 BM25 只看文档内容。但一个被大量网页链接的页面，通常质量更高。这就是 PageRank 的思想。

核心逻辑：

  • 网页 A 链接到网页 B → 相当于 A 给 B 投了一票
  • 来自高质量页面的链接权重更高
  • PageRank 是一个网页级别的全局评分，与查询词无关
  • d：阻尼因子（通常 0.85），表示用户有 85% 概率继续点击链接
  • Ti：所有链接到 A 的页面
  • L(Ti)：页面 Ti 的出链数

直觉理解：

想象用户随机点击链接浏览网页。PageRank 就是用户停留在某个页面的概率。被很多页面链接的页面，停留概率高；来自重要页面的链接，贡献更大。

Google 最初靠 PageRank 大幅提升搜索质量。但现代搜索引擎早已不只使用 PageRank，而是结合了数百种信号。

4.4 现代排序：多信号融合

2026 年的搜索引擎排序综合几百种信号：

信号类型 具体因素
内容相关性 BM25 分数、词频、位置、字段权重
页面质量 PageRank、链接数、域名权重
新鲜度 发布时间、更新频率
用户体验 点击率、停留时间、跳出率
上下文 用户位置、语言、搜索历史
页面体验 加载速度、移动端适配、HTTPS
结构化数据 Schema.org 标记
个性化 用户画像、兴趣偏好

最终排序通常用学习排序（Learning to Rank, LTR）：用机器学习模型综合所有信号给出最终分数。

4.5 向量检索与语义搜索

传统搜索是关键词匹配：搜”手机”找不到只写了”智能手机”的页面。

向量检索解决了这个问题：

  1. 用 Embedding 模型把每个文档转成一个向量（如 768 维）
  2. 用同样模型把查询转成向量
  3. 计算两个向量的余弦相似度，越接近越相关

即使文档中没有出现”手机”这个词，但向量空间中它们语义接近，仍能匹配。

5.2 去重

互联网上大量页面是复制粘贴的。搜索引擎需要对内容去重：

SimHash 算法：

  1. 对文档分词，每个词算一个哈希值
  2. 加权合并所有词的哈希值 → 得到一个 64 位指纹
  3. 两个文档的 SimHash 汉明距离 ≤ 3 → 判定为近似重复

SimHash 的优势：计算快，且相似文档的指纹也相似（不像 MD5 完全不同）。

5.3 数据更新策略

网页会变化，索引需要更新：

更新类型 策略 周期
新页面 爬虫发现后尽快索引 小时-天
已有页面更新 重新爬取，增量更新索引 天-周
页面删除 标记为已删除 天
索引重建 全量重建（极少） 月-年

Google 的 Caffeine 索引系统实现了近实时索引：新页面爬取后几秒内可被搜索到。
