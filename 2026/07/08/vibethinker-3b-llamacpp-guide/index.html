1.2 一句话定位

VibeThinker-3B 是一个 3B 参数的密集推理模型，专门针对数学竞赛、编程竞赛和 STEM 推理任务设计。它不是通用聊天模型，而是一个”偏科天才”——在可验证推理任务上能跟比自己大 200 倍的模型掰手腕，但在开放域知识问答上表现一般。

1.3 核心论点：参数压缩-覆盖假说

VibeThinker-3B 的技术报告提出了一个有意思的理论——Parametric Compression-Coverage Hypothesis（参数压缩-覆盖假说）：

不同能力对模型参数的依赖方式根本不同。可验证推理（数学、代码、STEM）是一种高度可压缩的、参数密集型能力——核心是多步推理、约束满足、自我纠错和答案验证。当任务空间足够结构化、反馈信号足够可靠时，小模型也能逼近前沿水平。相反，开放域知识和通用对话需要大规模参数来广泛覆盖事实和概念。

换句话说：数学和代码不需要”记住”很多事实，需要的是推理能力，而这可以被压缩到很小的模型里。

这个论点如果成立，对整个 AI 行业的含义是深远的——不是所有场景都需要堆参数。

2.2 这些数字意味着什么

AIME 2026 拿 94.3 分是什么概念？

AIME（American Invitational Mathematics Examination）是美国数学邀请赛，满分 15 题，每题 1 分，总分 15。94.3 分意味着在 100 分制下接近满分——这是一个极难的数学竞赛 benchmark。一个 3B 参数的模型跟 671B 的 DeepSeek V3.2 打平，这本身就是一件不可思议的事。

IMO-AnswerBench 76.4 分：

IMO-AnswerBench 包含 400 道国际数学奥林匹克级别的问题，是极具挑战性的 benchmark。VibeThinker-3B 以 3B 参数拿到 76.4，虽然略低于 GLM-5（82.5, 744B）和 Kimi K2.5（81.8, 1T），但差距远小于参数量差距。启用 CLR（Claim-Level Reliability Assessment）后提升到 80.6，已经进入第一梯队。

LeetCode 96.1% 通过率：

这是最让人震惊的数据。在 2026 年 4-5 月的 LeetCode 周赛中，VibeThinker-3B 首次提交通过了 128 道题中的 123 道。这意味着它不是在刷题库里的旧题——它真的在解从未见过的新题。

2.3 跟主流模型的对比定位

关键结论：在可验证推理这个维度上，VibeThinker-3B 实现了”参数效率”的跨越——用 1/224 的参数量达到了同等水平。

2.4 Sebastian Raschka 的评价

知名 AI 研究者 Sebastian Raschka 在博客中评价：

“这清楚地展示了好的数据筛选和后训练能带来多大提升。模型建立在较老的 Qwen2.5-Coder-3B 架构上，但通过精细的后训练管线达到了惊人效果。”

他根据 VibeThinker-1.5B 的成本估算，3B 版本的总训练成本大约在 $25,000 - $60,000 之间——不是百万美元级，但对小团队来说也不是小数目。

他也提了一个 caveat：模型很新，benchmark 可能过于乐观，需要实际使用验证。

三、技术架构：SSP 后训练管线

3.1 基座选择：为什么是 Qwen2.5-Coder-3B

VibeThinker-3B 没有从零训练，而是选择了 Qwen2.5-Coder-3B 作为基座。原因推测：

  1. 代码能力迁移：代码模型的逻辑推理能力可以迁移到数学推理
  2. 架构成熟：Qwen2 架构被广泛验证，llama.cpp/vLLM/SGLang 全线支持
  3. 参数效率：3B 是本地部署的甜点——手机能跑、树莓派能跑、任何笔记本都能跑
  4. MIT 友好：Qwen2.5 系列许可证宽松，适合二次开发

3.2 SSP 训练管线：四阶段

VibeThinker-3B 的核心创新在于后训练管线——Spectrum-to-Signal Principle (SSP)，从 1.5B 版本继承并升级。名字的意思是：先构建广泛（Spectrum）的推理轨迹，再放大正确信号（Signal）。

3.3 关键技术解析

MGPO（MaxEnt-Guided Policy Optimization）

这是 VibeThinker 的 RL 方法，属于 GRPO（Group Relative Policy Optimization）家族的变体。核心特点是最大熵引导——优先采样”不太容易也不太难”的样本，避免模型在已经掌握的简单题上浪费时间，也避免在完全不会的难题上空转。这个思路跟 curriculum learning 的精神一致。

Diversity-Exploring Distillation（多样性探索蒸馏）

在 SFT 阶段，不只保留一条正确解题路径，而是保留多条不同的有效路径。这让模型学会”一题多解”，增强推理的鲁棒性。

CLR（Claim-Level Reliability Assessment）

这是推理时的 test-time scaling 策略。不是简单的 majority voting，而是对推理过程中的每个 claim（论断）进行可靠性评估，根据可靠性加权得到最终答案。在 IMO-AnswerBench 上，CLR 把分数从 76.4 提升到 80.6。

64K 单窗口 RL

很多模型做 RL 时用渐进式上下文扩展（先短后长），VibeThinker-3B 直接用单一 64K 上下文窗口，保留完整的长时间推理轨迹。这避免了短窗口训练导致的”推理被截断”问题。

3.4 推理参数

官方推荐（benchmark 评估用 vLLM）：

参数 推荐值 说明
temperature 1.0 推理任务需要较高随机性探索不同路径
top_p 0.95 配合高 temperature
top_k -1（不限制） 让模型自由选择
max output tokens 最高 102K 复杂数学题可能需要超长推理链

注意：temperature 1.0 看起来很高，但对于推理模型来说是合理的——推理模型在 <think> 标签内做多步探索，需要随机性来尝试不同路径，最终答案在思考结束后给出，不受高 temperature 影响。

四、局限性：别拿它当万能模型用

4.1 官方明确声明的不适用场景

🚨 This model was not trained on tool-calling or agent-based programming data. We therefore do not recommend using it for tasks that involve function calling, API orchestration, or autonomous coding agents.

不擅长的事：

场景 原因
工具调用 / Function Calling 没训练过
Agent 编排 没训练过
自主编程 Agent 没训练过，建议只用于 LeetCode 式竞赛题
开放域知识问答 参数量限制，知识覆盖不足
通用聊天 不是设计目标，回答可能过于”推理化”
创意写作 不是设计目标

4.2 社区反馈

Reddit r/LocalLLaMA 社区的实际使用反馈：

  • 数学题确实强：”在 MathQA 上简直像巫术”——这是最常见的评价
  • 思考模式无法关闭：VibeThinker-3B 目前只支持 thinking mode，不像 Qwen3.5 那样可以切换 think/no_think。llama.cpp 可能因为 chat template 元数据问题标记为 non-thinking，但实际仍在思考
  • 思维链渲染问题：部分本地客户端（如 LM Studio）可能把 <think> 块内联显示，而不是单独折叠
  • benchmark 可能偏乐观：有用户反映实际体验不如 benchmark 那么惊艳，”benchmaxed really hard”

5.7 思维链处理

VibeThinker-3B 只支持 thinking mode——每次回答都会先输出 <think>...</think> 块，然后给出最终答案。这意味着：

  1. 输出更长：简单问题也可能有几百 token 的思维链
  2. 延迟更高：思维链生成时间 + 答案生成时间
  3. 需要解析：如果你想只展示最终答案，需要把 <think> 块分离出来

7.2 什么时候选 VibeThinker-3B

选它的情况：

  • 需要本地跑数学/编程推理，但硬件有限（3B 大小，1.93GB Q4）
  • 做教育产品，需要给学生展示详细解题过程
  • 搭建竞赛训练辅助工具
  • 研究小模型推理能力的前沿

不选它的情况：

  • 需要 Function Calling / Agent 能力 → 选 Qwen3.5-4B
  • 需要通用对话 → 选 Qwen3.5-4B 或 MiniCPM5-1B
  • 需要关闭 thinking 的快速响应 → 选 Qwen3.5-4B
  • 需要海量知识问答 → 选更大的模型（14B+）

Q1: llama.cpp 显示 non-thinking 模式怎么办

VibeThinker-3B 目前只支持 thinking 模式。如果 llama.cpp 标记为 non-thinking，这可能是 chat template 元数据问题，但模型实际仍在思考。检查方法：

Q3: 数学推理结果不正确

检查以下几点：

  1. 量化级别太低：Q3 以下可能损失推理精度，数学题建议 Q5_K_M 以上
  2. temperature 设置：推理任务用 temperature=1.0，不要用低温度
  3. 上下文不够：复杂题目需要 32K+ 上下文窗口
  4. 提示词不够明确：加上 “Think step by step and verify your answer”

总结

VibeThinker-3B 的意义不在于”又一个小模型”——而在于它证明了一件事：在可验证推理这个维度上，3B 参数就够了。

推理能力 同级别最强，甚至跨级别碾压
