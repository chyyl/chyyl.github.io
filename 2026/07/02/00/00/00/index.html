Ollama 流式输出踩坑：从乱码 SSE 到可控的文本/思维链/命令三通道

这是我做 OpenCloud Agent 时踩的最有意思的一个坑：Ollama 给你的流式输出是一堆逐 token 的碎片，你得自己拼回来，还得同时拆出三样东西——正文、思维链、JSON 命令。这篇文章记录我是怎么搞定的。

问题：Ollama 给了你什么

Ollama 的 /api/chat 接口开启 stream: true 后，返回的是 SSE 格式的逐行 JSON：

一个字一个字蹦。而且——Ollama SDK 不帮你做任何解析。<think> 标签？你自己拆。JSON 命令块？你自己检测。JSON 跨 chunk？你自己拼。

我需要从这堆碎片里分离出三个通道：

通道 用途 来源
text 给用户看的正文 模型输出的普通文本
thinking 思维链，CLI 里单独渲染 <think>...</think> 内的内容
command 工具调用命令 { "command": "...", "params": {...} }

第一关：拆 <think> 标签

朴素想法

正则一把梭：re.findall(r'<think>(.*?)</think>', full_text)。

问题：这是流式输出。你不能等全部文本到齐再拆——那跟非流式有什么区别？用户要的是逐字打字效果。

而且 <think> 标签的开闭可能被切到不同的 chunk 里：

正则在半个标签上根本匹配不到。

我的解法：状态机 + 末尾保留

核心思路是一个简单的状态机，两个状态：in_thinking = True/False。关键 trick 是：每次输出时检查末尾是否有未完成的标签前缀，如果有就留在 buffer 里不输出。

关键点：rfind("<th") 和 rfind("</") 这两个”末尾保留”逻辑。它们确保即使标签被切断，碎片留在 buffer 里等下一个 chunk 拼完整。

这个 pattern 后来我在很多流式处理场景中都用了——不只是标签，任何”标记可能跨 chunk”的场景都适用。

第二关：JSON 命令块检测

模型不只输出文字，还会输出工具调用命令：

这个命令块需要被检测、提取、解析成结构化数据传给工具执行器。

朴素想法

按行分割，找 { 开头的行，json.loads 解析。

问题：命令块经常跨 chunk。上面那个 JSON 可能是这样来的：

按行分？根本没换行。按 chunk 分？一个 chunk 不是一个完整 JSON。

我的解法：花括号深度计数 + 缓冲区

思路：一旦检测到 JSON 命令块的起始标记 {"command"，就进入”JSON 模式”，把后续所有 token 攒进 buffer，用花括号深度计数判断什么时候 JSON 完整了。

核心是 _try_extract_json()——逐字符扫描，跟踪花括号深度：

三个要注意的点：

  1. 字符串内的花括号不算。"params": {"query": "{test}"} 里的 {test} 不能影响深度计数。所以要跟踪 in_string 状态。
  2. 转义字符。"path": "C:\\folder" 里的 \\ 后面的引号不能翻转 in_string。所以要有 escape 标志。
  3. markdown 包裹。模型有时会把 JSON 包在 ```json ... ``` 里，解析前要剥掉。

第三关：两个解析器怎么串起来

<think> 拆分和 JSON 检测不能在同一步做——因为思维链里可能包含看起来像 JSON 的文本（模型在思考时可能写示例）。

我的架构是分两层：

model.py 先把流拆成 text 和 thinking 事件。parser.py 只处理 text 事件中的 JSON 命令块，thinking 事件直接透传。

这样思维链里的假 JSON 不会被误判为命令。

一个容易忽略的边界情况

模型输出 JSON 命令块之前，可能先输出一些正文文本。比如：

"好的，我来帮你搜索一下。\n\n" 这部分要作为 text 事件输出给用户，不能被吞掉。

但反过来，如果 buffer 末尾只有一个 {，它可能是 JSON 命令的开始，也可能是正文里的一个花括号。我的处理方式是：保留末尾最多 20 个字符中最后一个 { 之后的 内容，等下一个 chunk 来了再判断。如果后面跟着 "command"，进入 JSON 模式；如果没有，就当普通文本输出。

这个”等一等”的策略会让正文末尾有最多 20 字符的延迟，但对用户体验影响很小。

效果

最终的 CLI 渲染效果：

  • 正文实时逐字打出
  • 思维链在单独的面板里展示，灰色斜体
  • 命令块触发工具执行面板，显示工具名和参数

三通道并行，互不干扰。用户看到的是一个流畅的对话界面，背后是两个解析器在逐 token 地拆分流式数据。

总结

问题 朴素方案 为什么不行 我的方案
<think> 跨 chunk 正则匹配 半个标签匹配不到 状态机 + 末尾保留
JSON 跨 chunk 按行分割 命令块没有换行 花括号深度计数 + 缓冲区
思维链内的假 JSON 一起检测 误触发命令 分层解析，thinking 透传

核心教训就一句：流式数据不能等齐了再处理，但也不能盲目地按 chunk 处理。你需要一个缓冲区 + 一个状态机，让数据”流”过你的解析逻辑，在每个 token 到来时做增量判断。

这套东西写完之后，我发现它其实是一个很通用的 pattern。任何”结构化数据混在非结构化流里”的场景都能用：日志解析、协议拆包、混合内容渲染。AI Agent 只是触发我去想这个问题的契机。
