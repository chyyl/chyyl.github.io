Agent 循环设计：消息到工具执行的全链路拆解

大部分 AI 聊天是一问一答：用户发消息，模型回文字，结束。但真正的 Agent 不一样——模型可以自己决定调什么工具，拿到结果后继续推理，不够再调一次，直到它能给出最终答案。这篇文章拆解 OpenCloud Agent 的核心循环：消息怎么进去，工具怎么执行，结果怎么回传，循环怎么停。

全景：一轮循环长什么样

先看整体流程，再逐步拆解：

核心代码在 agent.py 的 chat() 方法里，196 行，我逐段拆。

第一步：组装消息列表

消息列表的结构是标准的 OpenAI 格式：system + history + 当前用户消息。self.conversation 是跨轮次持久化的对话历史，每次新消息都会追加进去。

注意一个细节：system prompt 不进 conversation。它是每次请求时临时拼上去的。这样做是因为 system prompt 可能会变（比如切换模型后换一套提示词），而对话历史应该是不可变的。

第二步：流式调用模型

这段是每轮的核心：把消息喂给 Ollama，拿回流式事件，用 StreamParser 拆成三个通道。

为什么用 StreamParser？ 因为 Ollama 的流式输出是逐 token 的碎片，文本、思维链、JSON 命令全混在一起。StreamParser 做的事就是实时把它们分开——文本直接 yield 给前端渲染，思维链单独走一个通道，JSON 命令块攒起来等流结束后统一执行。

这里有个设计选择：工具命令是等流结束后批量执行，而不是收到一个执行一个。 原因是模型可能在一次输出中给出多个命令（比如先搜索再读文件），批量执行更高效，也更容易做并行优化。

第三步：判断是否结束

如果这一轮模型没有输出任何 JSON 命令，说明它觉得答案已经完整了，直接把文本作为最终回复，跳出循环。

这是最简单的退出条件：模型不调工具 = 对话结束。

第四步：命令回注对话

如果有命令，需要把模型这一轮的完整输出（文本 + 思维链 + 命令）拼成 assistant 消息，加回 messages 列表：

为什么要拼回去？因为下一轮调用模型时，它需要知道”我上一轮说了什么、调了什么工具”。如果不把命令块回注，模型会失忆——它不知道自己刚才调过什么工具，可能会重复调用。

<think> 标签也要保留，因为模型的思维链是它推理过程的一部分。去掉思维链等于让模型每轮都从零开始想，上下文断裂。

第五步：执行工具

这段有三层逻辑，从上到下：

list 元工具

list 是特殊工具——它不是真正的工具调用，而是查询”有哪些工具/技能可用”。直接在 ToolRegistry 里同步处理，不走异步执行。这是因为 list 的结果不依赖外部环境，纯粹是本地元数据查询。

高风险操作拦截

这三个工具被标记为高风险。如果 confirm_high_risk=False，它们会被拦截，返回一个”需要用户确认”的错误结果。这个结果也会被回注给模型——模型会知道它的操作被阻止了，从而调整策略（比如改用其他方式，或者向用户解释为什么需要这个操作）。

正常执行

ToolRegistry 会根据命令名找到对应的异步 handler 执行。执行结果通过 yield 实时推给前端，同时也攒到 tool_results 里等会儿回注。

第六步：工具结果回注

这是整个循环最关键的一步。工具结果作为 user 消息 回注给模型，而不是 assistant 消息。

为什么用 user 角色？因为 Ollama 的 chat API 只认三种角色：system / user / assistant。工具结果不是模型说的（不能是 assistant），也不是系统指令（不能是 system），只能是 user。加 [工具执行结果] 前缀是为了让模型区分这是工具返回的数据，不是用户发的新消息。

回注之后，循环回到顶部，模型带着”我调了什么工具、工具返回了什么”的完整上下文，开始下一轮推理。

循环终止条件

两个退出条件：

  1. 模型不再输出命令 → 正常结束，文本作为最终回复
  2. 达到 10 轮上限 → 强制终止，返回提示信息

为什么是 10 轮？这是经验值。大部分任务 1-3 轮就够：搜索→读文件→回答。复杂任务（多步搜索+交叉验证）可能需要 5-7 轮。10 轮给够余量，同时防止模型陷入无限循环——比如它反复调同一个工具拿不到满意的结果，或者陷入”调工具→结果不对→再调工具→还是不对”的死循环。

实际上，如果模型真的跑到第 10 轮还没出来，通常是提示词设计有问题，或者任务超出了模型的能力范围。这个上限更像是一个安全网。

事件系统：前端怎么消费

整个 chat() 是一个异步生成器，通过 yield 把事件实时推给前端：

事件类型 含义 前端处理
text 模型文本片段 逐字渲染到聊天气泡
thinking 思维链片段 灰色斜体面板，可折叠
tool_call 工具开始执行 显示工具名+参数卡片
tool_result 工具返回 更新工具卡片状态
final 最终回复 标记对话完成
error 错误信息 红色提示
info 系统信息 灰色提示

CLI 用 Rich 的 Live 渲染这些事件，Web 端通过 WebSocket 推给浏览器。前端不需要知道循环逻辑，只需要按事件类型渲染——关注点分离。

一个完整的例子

用户问：”帮我看看当前目录有哪些 Python 文件，然后统计一共多少行代码。”

第 1 轮：

  • 模型输出：{"command": "list_files", "params": {"path": ".", "pattern": "*.py"}}
  • 工具执行：返回 ["main.py", "agent.py", "parser.py", ...]
  • 结果回注给模型

第 2 轮：

  • 模型输出：{"command": "read_file", "params": {"path": "main.py"}} × N 个文件
  • 工具执行：返回每个文件的内容
  • 结果回注

第 3 轮：

  • 模型输出："当前目录有 20 个 Python 文件，共计 3500 行代码。其中 agent.py 最大（196 行），parser.py 次之（151 行）。"
  • 没有命令 → 循环结束

三轮，刚好够用。

设计取舍总结

决策 选择 理由
命令执行时机 批量执行 支持多命令、未来可并行
工具结果角色 user API 限制 + 语义合理
轮数上限 10 经验值，防死循环
system prompt 不进 conversation 可变，不持久化
高风险操作 拦截+回注 模型知道被拦了，能调整策略
事件推送 异步生成器 流式渲染，关注点分离

核心设计理念就一句：Agent 循环不是一问一答，而是”推理→行动→观察→再推理”的迭代过程。 每一轮都是模型基于完整上下文做决策，工具结果作为新信息注入，推动模型向最终答案收敛。

这个 pattern 不只适用于 AI Agent。任何”决策器+执行器+反馈”的系统都能用：自动化运维、游戏 AI、机器人控制。AI Agent 只是我触发去想这个问题的契机——跟流式解析那篇一样。
